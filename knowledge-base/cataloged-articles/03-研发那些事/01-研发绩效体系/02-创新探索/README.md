# 02-创新探索

这个模块讨论 0->1 阶段如何做创新、如何评价探索质量。

核心判断：

**创新不是自由发挥，而是在不确定性中提出假设、验证假设、形成结论，并推动问题收敛。**

当前文章：

| 文件 | 说明 |
|------|------|
| 03-如何做创新探索.md | 讨论创新探索的结构、评价方式和收敛机制 |

## 适合展开的主题

这个模块用“问题 -> 假设 -> 验证 -> 决策 -> 沉淀”来拆。它覆盖创新探索的完整闭环，也能和工程交付模块分开：这里重点是降低方向不确定性，不讨论规模化交付。

| 分类 | 可写主题 | 说明 |
|------|----------|------|
| 问题 | `如何判断一个问题值得探索.md` | 判断问题真实性、重要性、窗口期和投入价值 |
| 假设 | `如何定义关键假设.md` | 把模糊想法拆成可验证假设 |
| 验证 | `探索项目如何设计验证.md` | 讨论 PoC、原型、实验、访谈、技术预研等验证方式 |
| 验证 | `技术预研和产品创新的区别.md` | 区分技术可行性验证和用户价值验证 |
| 决策 | `创新失败如何评价.md` | 区分高质量失败和低质量失败 |
| 决策 | `探索项目的退出标准.md` | 定义 Go / No-Go / Pivot / Hold |
| 沉淀 | `探索结果如何转入工程交付.md` | 把探索结论、风险和约束交接给交付阶段 |

## 不放在本模块的内容

1. 交付质量、稳定性、技术债治理放到 `03-工程交付`；
2. KPI 的通用设计放到 `01-KPI与研发绩效`；
3. 企业案例和初创团队实践放到 `04-企业类型与初创团队`。
