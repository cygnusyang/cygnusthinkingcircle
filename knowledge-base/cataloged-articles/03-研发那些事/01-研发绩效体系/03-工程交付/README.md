# 03-工程交付

这个模块讨论研发从验证走向可靠交付时，应该如何管理绩效。

核心判断：

**工程交付不是把功能做完，而是把经过验证的方案做成可靠、可维护、可复用、可规模化的系统能力。**

当前文章：

| 文件 | 说明 |
|------|------|
| 04-如何做工程交付.md | 讨论工程交付的定义、质量、技术债和绩效评价 |

## 适合展开的主题

这个模块用“交付定义 -> 质量 -> 效率 -> 技术债 -> 稳定性 -> 复用沉淀”来拆。它覆盖从可行方案到可靠系统的工程化过程，重点是降低工程不确定性。

| 分类 | 可写主题 | 说明 |
|------|----------|------|
| 交付定义 | `什么叫真正交付完成.md` | 区分开发完成、测试完成、上线完成和真正交付完成 |
| 质量 | `研发质量如何进入绩效.md` | 讨论缺陷、测试覆盖、质量门禁、事故复盘 |
| 效率 | `工程效率指标怎么设计.md` | 讨论交付周期、部署频率、构建耗时、阻塞关闭周期 |
| 技术债 | `技术债如何评价和治理.md` | 讨论可接受技术债、不可见技术债和恶性技术债 |
| 稳定性 | `稳定性和可靠性如何评价.md` | 讨论可用性、回滚、降级、监控告警和故障恢复 |
| 复用沉淀 | `交付如何形成长期工程能力.md` | 讨论组件化、平台化、文档、自动化和知识扩散 |

## 不放在本模块的内容

1. 探索期的假设验证放到 `02-创新探索`；
2. 价值观和低绩效判断放到 `06-组织与价值观`；
3. 一页纸模板和评审表放到 `05-框架与模板`。
