# 00-总纲

这个模块回答两个基础问题：

1. 研发到底是什么；
2. 什么样的研发工作应该被认为是高绩效。

当前文章：

| 文件 | 说明 |
|------|------|
| 01-研发绩效底层逻辑.md | 解释研发绩效的底层逻辑：方向、节奏、风险、长期能力 |

## 适合展开的主题

这个模块用“定义层 -> 逻辑层 -> 边界层 -> 关系层”来拆，尽量满足 MECE：先定义对象，再解释逻辑，再划清边界，最后说明它和外部概念的关系。

| 分类 | 可写主题 | 说明 |
|------|----------|------|
| 定义层 | `研发到底是什么.md` | 定义研发不是简单任务执行，而是不确定性转确定性的过程 |
| 定义层 | `研发绩效到底是什么.md` | 定义研发绩效评价的对象和范围 |
| 逻辑层 | `研发绩效为什么不能只看结果.md` | 解释方向、节奏、风险、长期能力之间的关系 |
| 逻辑层 | `研发绩效的阶段模型.md` | 解释 0->1、1->10、1->100 的差异 |
| 边界层 | `研发组织为什么不能只用交付率评价.md` | 说明研发绩效和交付数量之间的边界 |
| 关系层 | `研发绩效和业务绩效的关系.md` | 说明研发结果如何支撑业务结果，但两者不能简单等同 |

## 不放在本模块的内容

1. KPI 指标设计放到 `01-KPI与研发绩效`；
2. 创新探索的具体方法放到 `02-创新探索`；
3. 工程交付和质量管理放到 `03-工程交付`；
4. 模板和工具放到 `05-框架与模板`。
